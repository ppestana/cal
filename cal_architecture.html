Criticar a Arbitragem Legalmente

01

01 — Visão Geral

O CAL é um sistema de data analytics que quantifica desvios entre decisões de arbitragem esperadas e decisões observadas. Para cada árbitro, temporada e tipo de decisão, o sistema calcula Z-scores de viés e um score composto de suspeição que permite comparação e ranking entre árbitros.

Âmbito actual: Primeira Liga Portuguesa (Liga Portugal Betclic), temporadas 2017/18 a 2025/26. Dados agregados por jogo via football-data.co.uk (golos, cartões, faltas) e Sofascore API (árbitros). Totalmente gratuito.
Expansão possível: O modelo xP (expected penalty) completo requer contact_type, location_x/y e defender_position disponíveis via Sportmonks (~€50/mês). Fora do âmbito actual — os modelos de viés para cartões e faltas são totalmente viáveis com dados gratuitos.

06 — Modelos Estatísticos

✓ Fase 3 completa

Três modelos estimam a probabilidade esperada de cada decisão de arbitragem dado o contexto do jogo. Treinados com leave-one-season-out cross-validation em 4816 observações (jogos × equipas). Modelos serializados em cal/models/saved/models.joblib.

Regressão Logística — xYC

P(cartão amarelo | contexto). Brier score médio: 0.067. Features reais implementadas.

xYC ~ is_home + matchday + team_points_r10 + opp_points_r10 + ref_yellows_pg_r20 + ref_fouls_pg_r20 + ref_games_total

Regressão Logística — xRC

P(cartão vermelho | contexto). Brier score médio: 0.113. Evento raro (~13% dos casos).

xRC ~ is_home + matchday + team_points_r10 + opp_points_r10 + ref_reds_pg_r20 + ref_yellows_pg_r20 + ref_games_total

Modelo Poisson — xF (faltas)

E(faltas | contexto). MAE médio: 3.3 faltas/jogo. Captura tendência descendente 2017→2025.

log(xF) = is_home + matchday + team_points_r10 + team_goals_scored_r10 + team_goals_conceded_r10 + ref_fouls_pg_r20

Validação — LOSO Cross-Validation

Leave-One-Season-Out: treinar em 7 épocas, validar na 8ª. Repete para cada época. Garante que os bias scores são calculados com modelos que nunca viram os jogos avaliados.

para cada época T
treinar em épocas ≠ T
prever em época T
guardar em expected_probabilities

07 — Bias Engine

✓ Fase 4 completa

Quatro camadas de análise que, em conjunto, constroem a evidência estatística de padrões anómalos na arbitragem. A combinação das quatro é necessária para distinguir anomalia isolada de padrão sistemático. 

# Camada 1 — Z-scores por árbitro+época (binomial Z-test)
Z_yellows = (obs_jogos_com_amarelo - exp_yellows) / sqrt(exp * (1 - exp/n))
Z_fouls   = (obs_mean_fouls - exp_mean_fouls) / (std_fouls / sqrt(n))
SuspicionScore = |Z_yellows| + |Z_reds| + |Z_fouls|

# Camada 2 — Consistência temporal (multi-época)
evidence_score = (n_épocas_biased_y × consistency_y) + (n_épocas_biased_f × consistency_f)

# Camada 3 — Interacção árbitro×equipa (regressão logística)
P(amarelo) ~ is_home + team_effect + referee × team   # p_value < 0.05 = significativo

# Camada 4 — Cartões por equipa (comparação directa)
desvio = media_amarelos_equipa - media_geral_arbitro   # > +0.8 = desvio extremo
Z-score / SuspicionScore Interpretação Acção
|Z| < 1.5 Normal — dentro da variância esperada Nenhuma
1.5 ≤ |Z| < 2.5 Ligeiramente atípico Monitorizar
2.5 ≤ |Z| < 4 Suspeito — desvio estatisticamente relevante Análise aprofundada
|Z| ≥ 4 ou consistência multi-época Padrão anómalo forte Alerta
Nota metodológica: Z-scores elevados indicam padrões estatisticamente anómalos, não prova de parcialidade intencional. A evidência mais robusta é a combinação de: (1) Z-score elevado, (2) padrão consistente em múltiplas épocas, (3) desvio significativo face aos peers. Um árbitro que aparece nas três análises simultaneamente é o caso mais preocupante.

Pipeline incremental (após cada jornada)

Forma mais simples — botão no dashboard: Na sidebar do dashboard Streamlit (http://localhost:8501), secção 🔄 Actualizar dados, confirmar a época (ex: 2025/26) e carregar ▶ Actualizar agora. O dashboard corre os 5 scripts automaticamente e recarrega com os dados novos. O botão é idempotente — pode ser carregado quantas vezes for necessário sem duplicar dados.
Fluxo típico por jornada:
Domingo à noite — carregar o botão depois dos jogos do fim de semana.
Segunda-feira — carregar de novo após o último jogo da jornada (jogos em atraso).
A pipeline detecta automaticamente os jogos novos e actualiza só o necessário.

Alternativa via CLI (se o dashboard não estiver a correr): 

# Actualizar apenas a época em curso — não re-processar épocas passadas
docker compose exec dev python run_ingest.py "2025/26"
docker compose exec dev python run_referees.py "2025/26"
docker compose exec dev python run_features.py "2025/26"
docker compose exec dev python run_bias.py
docker compose exec dev python run_cards.py
Fim de época — re-treinar modelos: No fim de cada época, após a ingestão completa, re-treinar os modelos com todos os dados disponíveis. Este passo melhora a precisão dos bias scores para a época seguinte.

docker compose exec dev python run_models.py

Isto actualiza expected_probabilities e, por consequência, todos os bias scores. Depois correr novamente run_bias.py e run_cards.py para recalcular com as novas predições.

2026-03-15
v0.9.1

Fases 2, 3 e 4 — Feature Engineering, Modelos e Bias Engine

Feature matrix completa (match_features, 4896 linhas, 21 colunas) com rolling team strength e rolling referee stats. Três modelos: xYC (Brier 0.067), xRC (Brier 0.113), xF Poisson (MAE 3.3). Bias engine: Z-scores binomial + t-test, consistência temporal, peer comparison, interacção árbitro×equipa. Análise de cartões por árbitro×equipa×época — 39 desvios extremos identificados. Sofascore API para árbitros (98.4% cobertura). Docker portátil operacional. Épocas 2025/26 adicionadas.
